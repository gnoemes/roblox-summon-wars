## 1) Архитектура Experience
**В опыте ровно 2 place:**
1) **SOCIAL HUB** — публичный хаб (много серверов, игроки видят друг друга).
2) **RAID PLACE** — отдельный place для рейдов в **приватных reserved servers** (соло/пати 1–4).

Хаб = социалка и сервисы. Рейд = изолированная боевая сессия.

---

## 2) Social Hub (place)

### 2.1 Визуальная концепция
- Платформа/станция **в космосе под куполом**.
- Skybox глобально = космос; “ощущение купола” делается геометрией, локальным освещением, эффектами и декором.
- Дом остаётся как часть сцены (и будущих фич), без “закрытых комнат”.

### 2.2 Функциональные зоны (полная версия)
- **Raid Console / “Гильдия”**: создать рейд, открыть/закрыть лобби, собрать пати.
- **Лаборатория**: мутации (панель/колба), бридинг (3 колбы: 2 родителя + результат).
- **NPC Pet Market**: покупка питомцев у NPC (каталог/ротация).
- **Shop (soft currency)**: расходники/базовые материалы.
- **Дом**: витрина/трофеи/будущая система.
- **Социальные элементы**: доска дейли, лидерборды, зоны общения.

### 2.3 Что в MVP, что позже
**MVP:**
- Raid Console (пати + старт рейда)
- Лаборатория (мутации+бридинг)
- NPC Pet Market
- Shop за игровую валюту
- Мини-питомец follower + inspect карточка (социальный крючок)

**Позже:**
- Кросс-серверный LFG (межсерверный поиск лобби)
- Торговля между игроками (сначала pet-for-pet)
- Дом как система (трофеи/посетители)
- Парк/пруд/рыбалка/редкая еда

## 3) Raid place (private reserved servers)

### 3.1 Основной формат
- Сессионный рейд: **только пати** (1–4), никто посторонний не подключается.
- Игрок переходит в “режим командования” (HUD/камера/оверлей), управляет питомцем командами.
- Арены: плоскость/площадки, спавны по краям, волны мобов.
- Для босса награда только при убийстве.

---

## 4) Teleport flow (Hub → Raid → Hub)

### 4.1 Create raid session (Hub server)
1) Лидер выбирает: режим (Waves/Boss), сложность, модификаторы, арена.
2) Hub server создаёт `SessionId`.
3) Hub server резервирует сервер рейда: `ReserveServerAsync(raidPlaceId)` → `accessCode`.
4) Hub server создаёт `SessionRecord` и кладёт в **MemoryStore** с TTL (10–30 минут).
5) Hub server телепортит игроков в reserved server (`TeleportToPrivateServer` / через `TeleportAsync`).

### 4.2 Init raid (Raid server)
1) Raid server читает TeleportData (минимум `SessionId`).
2) Поднимает `SessionRecord` из MemoryStore.
3) Валидирует: party, режим, конфиг.
4) Если невалидно/нет записи — возврат в хаб.

### 4.3 End raid (Raid server)
1) Сервер рейда рассчитывает прогресс/результат, выдаёт награды по правилам (см. Economy_Progression).
2) Записывает итог в персистент (DataStore/профиль).
3) Телепортирует игроков обратно в Hub place (TeleportData может содержать summary).

---

## 5) Сессии, данные, TTL

### 5.1 SessionRecord (минимум)
- `SessionId`
- `RaidConfig` (mode, difficulty, modifiers, arenaId, seed)
- `Party` (leaderUserId, userIds[])
- `CreatedAt`, `ExpiresAt`
- `RaidAccessCode` (если нужен rejoin в будущем)

### 5.2 Политика дисконнектов
- Награды, уже выплаченные на milestone, сохраняются.
- Дисконнект: **не выдаёт** ничего “за текущий сегмент”.

## 6) Party & LFG (поиск лобби)

### 6.1 MVP: внутри одного hub-сервера
- Лидер открывает лобби → приглашает/принимает → ready → start.

### 6.2 Позже: межсерверный LFG (глобальная доска)
Глобальный список лобби через `MemoryStoreService` (SortedMap) + атомарные апдейты слотов.

**Храним:** `LobbyId -> { leaderUserId, hubJobId, desiredPartySize, openSlots, config, createdAt, expiresAt }`

**Join flow:**
1) Сервер делает `UpdateAsync` записи лобби (занять слот атомарно).
2) Затем:
   - **Вариант A:** телепорт игрока в **тот же hub instance** лидера через `TeleportToPlaceInstance(hubPlaceId, hubJobId, player)`; партия собирается у лидера и стартует рейд.
   - **Вариант B:** лобби заранее резервирует raid server и публикует access code; игрок телепортируется напрямую в raid reserved server.

---

## 7) Anti-exploit принципы
- Все награды и изменения профиля фиксирует **сервер рейда**.
- Клиент может показывать “дропы”, но это визуал — не источник правды.
- Любые команды питомца валидируются на сервере.

## Content topology note (обновлено 2026-03-06)
На текущем этапе рекомендуется держать:
- **1 Hub place**
- **1 Raid place**

Все биомы, tier'ы, модификаторы, сценарии и награды подгружаются **внутри одного Raid place** через конфиги и пресеты.

Это уменьшает:
- сложность телепортов
- дублирование логики
- стоимость выпуска нового контента
